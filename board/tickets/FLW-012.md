# FLW-012 — Pass `flow_id` from frontend run trigger to `POST /hedge-fund/run`

**Status:** Ready for dev
**Priority:** Must
**MoSCoW rationale:** FLW-009 adds the persistence write path on the backend, but it only fires when `flow_id` is present in the request body. Without this ticket, the frontend never sends `flow_id` and the DB write never happens.
**Depends on:** FLW-008 (flow_id accepted by backend schema)
**Blocks:** Nothing (can be implemented in parallel with FLW-009, FLW-010, FLW-011; the full feature requires all five)
**Spec refs:** `docs/requirements/flow-run-persistence.md` — AC-0243, AC-0244

---

## Problem Statement

When the user clicks "Run" in the portfolio-start-node, `runFlow()` in `use-flow-connection.ts` is called with a `HedgeFundRequest` params object. That object (constructed in `portfolio-start-node.tsx` lines 235–254) does not include `flow_id`. The `flowId` string is already available inside `use-flow-connection.ts` (it is the hook's primary input), but is never forwarded to the API params before the `fetch` call in `api.ts`. This ticket adds that forwarding.

## Implementation Notes

### 1. `use-flow-connection.ts` — inject `flow_id` before the API call

`runFlow` is defined at line 115. It receives `params: HedgeFundRequest`. The hook already has `flowId` (a `string | null`) in scope. Before calling `api.runHedgeFund(params, nodeContext, flowId)`, set the `flow_id` field:

```typescript
const runFlow = useCallback((params: HedgeFundRequest) => {
  if (!flowId || !canRun) return;

  // Attach the current flow's DB ID so the backend can persist the run
  const flowIdInt = flowId ? parseInt(flowId, 10) : null;
  if (flowIdInt && !isNaN(flowIdInt)) {
    params.flow_id = flowIdInt;
  }

  // Reset node states for this flow
  nodeContext.resetAllNodes(flowId);
  // ... rest of existing code unchanged ...
```

The mutation of `params` here is safe: `params` is a freshly-constructed literal object passed in from `portfolio-start-node.tsx` on each invocation — it is not a stable reference held elsewhere.

### 2. `HedgeFundRequest` TypeScript type in the frontend

The frontend likely has its own `HedgeFundRequest` interface defined in `api.ts` or inferred from usage. Check for a TypeScript definition of `HedgeFundRequest` in `app/frontend/src/services/api.ts`. If `HedgeFundRequest` is typed there, add `flow_id?: number | null` to it. If the params are typed as a plain object literal (no named interface), add the field directly to the object literal in the `runFlow` call site.

Search for the interface definition:

```
grep -n "HedgeFundRequest" app/frontend/src/services/api.ts
```

If an `interface HedgeFundRequest` exists in the frontend, add `flow_id?: number | null;` to it. If not, the mutation approach above is sufficient and TypeScript will accept it because `params` is already `any`-adjacent in usage.

### 3. `api.ts` — confirm `flow_id` passes through

In `api.ts` at line 101: `const backendParams = params;`. This is a reference copy — `backendParams` is the same object as `params`. On line 114: `body: JSON.stringify(backendParams)`. Since `params.flow_id` was set in step 1 above, it will be serialized. No structural change to `api.ts` is needed beyond confirming the TypeScript type (step 2 above).

### 4. `portfolio-start-node.tsx` — no change required

The `runFlow()` call at line 235 does not need to change. The `flowId` injection happens inside `use-flow-connection.ts`, not at the call site. This keeps the portal-start-node dumb about persistence concerns.
