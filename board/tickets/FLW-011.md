# FLW-011 — Restore last COMPLETE run results in `flow-context.tsx::loadFlow()`

**Status:** Ready for dev (depends on FLW-010)
**Priority:** Must
**MoSCoW rationale:** This is the user-visible restore path. Without it, the DB persistence from FLW-009 is write-only and the founder still loses results on refresh.
**Depends on:** FLW-010 (getLatestFlowRun and FlowRunResponse type)
**Blocks:** Nothing
**Spec refs:** `docs/requirements/flow-run-persistence.md` — AC-0245, AC-0246, AC-0247, AC-0248

---

## Problem Statement

`flow-context.tsx::loadFlow()` restores nodes, edges, and viewport from the saved flow but makes no call to load prior run results. `nodeContext.setOutputNodeData()` is the mechanism that renders results in the output node — it is called by `api.ts` during a live run (line 197) but never called during flow load. This ticket adds that call after a successful flow load.

## Implementation Notes

### 1. Add import

`flowService` is already imported at line 5:
```typescript
import { flowService } from '@/services/flow-service';
```

Add `FlowRunResponse` to that same import line:
```typescript
import { flowService, FlowRunResponse } from '@/services/flow-service';
```

`nodeContext` is NOT currently imported in `flow-context.tsx`. It must be added. Look at how `api.ts` uses it — `nodeContext` is passed as a parameter from `use-flow-connection.ts`. In `flow-context.tsx` the pattern is different: the context provider does not receive `nodeContext` as a prop today.

Two options exist for accessing `nodeContext` inside `flow-context.tsx`:

**Option A (recommended):** Use `useNodeContext()` hook directly inside `FlowProvider`. The `NodeContextProvider` wraps `FlowProvider` in the component tree (verify in `app.tsx` or the root layout). If `FlowProvider` is rendered inside `NodeContextProvider`, calling `useNodeContext()` inside `FlowProvider` is valid.

**Option B (fallback):** Pass the restore data up to the caller and let the caller call `nodeContext.setOutputNodeData()`. This is more invasive — `loadFlow()` would need to return the restored data, and every call site would need to handle it.

Assume Option A and verify the component tree before implementing. If `FlowProvider` is not inside `NodeContextProvider`, escalate to the founder to decide the nesting order.

### 2. Add restore logic inside `loadFlow()`

The `loadFlow` callback currently ends at line 184 (before the closing `}` of the try block). Insert the following after the `setTimeout` block for connection state recovery (after line 184) and before the closing `}` of the try block:

```typescript
// Restore the most recent COMPLETE run results into the output node
try {
  const latestRun: FlowRunResponse | null = await flowService.getLatestFlowRun(flow.id);
  if (latestRun && latestRun.status === 'COMPLETE' && latestRun.results) {
    nodeContext.setOutputNodeData(flow.id.toString(), latestRun.results as any);
  }
} catch (restoreError) {
  // Non-fatal: log and continue. Flow loads normally without results.
  console.warn('Could not restore last run results:', restoreError);
}
```

The `as any` cast on `latestRun.results` is acceptable here because `setOutputNodeData` accepts `OutputNodeData` and the `results` JSON stored in the DB is exactly the payload produced by the `complete` SSE event, which is `OutputNodeData`. This matches AC-0247 (identical rendering to live run). If the TypeScript shape can be tightened later, that is a follow-up.

### 3. The `useFlowContext` hook does NOT need to change

`FlowContextType` interface and the `value` object do not need a new field. The restore is a side-effect inside `loadFlow()`, not a new public API.

### 4. Ordering within `loadFlow()`

The restore call must happen after `reactFlowInstance.setNodes()` and `reactFlowInstance.setEdges()` (lines 158–159) so that the output node component is mounting at the same time as (or before) `setOutputNodeData` is called. The existing `setTimeout` at line 177 handles the connection-state check. The restore call can follow the `setTimeout` block but does not need to be inside it — `setOutputNodeData` writes to React state and React batches the update naturally.

## Acceptance Criteria

**AC-0245** — After `loadFlow()` completes, if the flow has a COMPLETE run with non-null `results`, the output node renders the analyst signals and portfolio decisions identically to how they appear after a live run.

**AC-0246** — If `getLatestFlowRun` returns `null`, returns a non-COMPLETE run, or throws, `loadFlow()` completes normally with no error surfaced in the UI and the canvas loads with a blank output node.

**AC-0247** — The restored output is visually indistinguishable from a freshly-completed run's output. No label like "Restored" or "Stale" is shown (this is intentional — the data is the same data).

**AC-0248** — After restoration, the run button is in its normal idle state. The user can trigger a new run immediately. (This is the existing behavior since `flowConnectionManager` state is `idle` on load — the restore adds only data, not a connected/processing state.)
