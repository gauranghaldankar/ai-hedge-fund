# FLW-008 — Add optional `flow_id` to `HedgeFundRequest` schema

**Status:** Ready for dev
**Priority:** Must
**MoSCoW rationale:** Every other ticket in this feature depends on the backend receiving a `flow_id`. Without it, the run endpoint has no way to associate an execution with a saved flow, making persistence impossible.
**Depends on:** Nothing
**Blocks:** FLW-009
**Spec refs:** `docs/requirements/flow-run-persistence.md` — AC-0237, AC-0238

---

## Problem Statement

`HedgeFundRequest` (the Pydantic model for `POST /hedge-fund/run`) has no `flow_id` field. The run endpoint cannot associate an execution with a `HedgeFundFlow` DB row, so `FlowRunRepository` can never be called. Adding the field as an optional int preserves full backward compatibility with any call site that omits it.

---

## Files to Change

| File | Line guidance |
|---|---|
| `app/backend/models/schemas.py` | Add `flow_id: Optional[int] = None` to `BaseHedgeFundRequest` (line 61–70). Since `HedgeFundRequest` and `BacktestRequest` both inherit from `BaseHedgeFundRequest`, both automatically pick up the field. |

Only one file changes. No migration, no new table.

---

## Implementation Notes

Add to `BaseHedgeFundRequest` at line 70 (after `api_keys`):

```python
flow_id: Optional[int] = None  # DB ID of the HedgeFundFlow this run belongs to
```

`Optional[int]` is already imported at the top of the file (`from typing import List, Optional, Dict, Any`). No import change required.

This field is declared on `BaseHedgeFundRequest` so that future work can also persist backtest runs by reading `request.flow_id` from `BacktestRequest` without a second schema change.

Do NOT add any validation logic (e.g., checking the flow exists). That check belongs in the run endpoint handler (FLW-009), not the schema.

## Acceptance Criteria

**AC-0237 (partial)** — Existing unit tests or API call sites that do not pass `flow_id` continue to work without modification.
